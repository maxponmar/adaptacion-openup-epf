Task Descriptor: Run Tests
Ejecutar los scripts de pruebas adecuados, analizar los resultados, describir problemas y comunicar los resultados de las pruebas al equipo.

Steps
Revisar los elementos de trabajo completados en la construcción.
Revisar los elementos de trabajo que se integraron en la construcción desde el último ciclo de prueba. Enfocarse en identificar requisitos previamente no implementados o que antes fallaban y que ahora se espera que cumplan las condiciones de satisfacción.
Seleccionar Scripts de Prueba.

Seleccionar los scripts de prueba relacionados con los elementos de trabajo completados en la construcción.

Idealmente, cada ciclo de prueba debería ejecutar todos los scripts de prueba, pero algunos tipos de pruebas requieren demasiado tiempo para incluirse en cada ciclo de prueba. Para pruebas manuales o que consumen mucho tiempo, incluir los scripts de prueba que proporcionarán la retroalimentación más útil sobre la solución en desarrollo en función de los objetivos de la iteración.

Planificar con conjuntos de pruebas para simplificar el proceso de selección de pruebas para cada construcción (consultar la Guía: Conjunto de Pruebas).

Ejecutar los scripts de prueba contra la construcción.

Ejecutar las pruebas siguiendo el procedimiento paso a paso en el Script de Pruebas.

Para los scripts de pruebas automatizados, iniciar la ejecución de la prueba. Los scripts de pruebas automatizados deben ejecutarse en conjuntos en la secuencia correcta y recopilar resultados en el Registro de Pruebas.

Para ejecutar un script de prueba manual, establecer sus condiciones previas, realizar los pasos mientras se registran los resultados en el Registro de Pruebas y llevar a cabo cualquier paso de limpieza (teardown) necesario.

Analizar y comunicar los resultados de las pruebas.

Publicar los resultados de las pruebas en un lugar destacado y accesible para todo el equipo, como un pizarrón blanco o una wiki.

Para cada script de prueba que falle, analizar el Registro de Pruebas para identificar la causa del fallo en la prueba. Comenzar con las pruebas que fallaron y que se esperaba que comenzaran a pasar en esta construcción, lo que puede indicar elementos de trabajo recién entregados que no cumplen con las condiciones de satisfacción. Luego, revisar los scripts de prueba que anteriormente pasaban y que ahora fallan, lo que puede indicar problemas regresivos en la construcción.

Si una prueba falló porque la solución no cumple con las condiciones de satisfacción del caso de prueba, registrar el problema en la Lista de Elementos de Trabajo. En el elemento de trabajo, identificar claramente el comportamiento observado, el comportamiento esperado y los pasos para repetir el problema. Indicar qué prueba fallida descubrió inicialmente el problema.

Si una prueba falló debido a un cambio en el sistema (como un cambio en la interfaz de usuario), pero la implementación aún cumple con las condiciones de satisfacción del caso de prueba, actualizar el script de prueba para que pase con la nueva implementación.

Si una prueba falló porque el script de prueba es incorrecto (un resultado negativo falso) o pasó cuando se esperaba que fallara (un resultado positivo falso), actualizar el script de prueba para implementar correctamente las condiciones de satisfacción del caso de prueba. Si el caso de prueba para un requisito es inválido, realizar la Tarea: Solicitar Cambio para modificar las condiciones de satisfacción del requisito.

Es mejor actualizar los scripts de prueba lo más rápido y continuamente posible (consultar Tarea: Implementar Scripts de Prueba). Si el cambio en el script de prueba es trivial, actualizar la prueba mientras se analizan los resultados de las pruebas. Si el cambio es una tarea no trivial, enviarlo a la Lista de Elementos de Trabajo para que se priorice frente a otras tareas.

Proporcionar retroalimentación al equipo.

Resumir y proporcionar retroalimentación al equipo sobre qué tan bien la construcción satisface los requisitos planificados para la iteración. Enfocarse en medir el progreso en términos de pruebas que pasan.

Explicar los resultados del ciclo de pruebas en el contexto de tendencias generales:

- ¿Cuántas pruebas se seleccionaron para la construcción y cuál es su estado (aprobado, fallado, bloqueado, no ejecutado, etc.)?
- ¿Cuántos problemas se agregaron a la Lista de Elementos de Trabajo y cuál es su estado y gravedad?
- Para los scripts de prueba que fueron bloqueados o omitidos, ¿cuáles son las razones principales (como problemas conocidos)?
