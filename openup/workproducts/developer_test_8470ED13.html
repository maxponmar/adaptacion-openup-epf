Description
Main Description

Este artefacto abarca todos los pasos necesarios para validar un componente de software. Especifica las entradas de prueba, las condiciones de ejecución y los resultados esperados. Estos detalles se identifican con el propósito de evaluar un aspecto particular de un escenario.

Las pruebas deben ser autoexplicativas de manera que quede claro al finalizar la prueba si el componente se ha ejecutado correctamente.


Posibles pruebas por implementar:

  • Pruebas de Funcionalidad:

Verificar que todas las funciones principales de la plataforma funcionen correctamente, como la carga de tesis, el análisis de riqueza léxica y conectividad, y la generación de retroalimentación.

  • Pruebas de Interfaz de Usuario (UI):

Evaluar la usabilidad y la experiencia del usuario al interactuar con la plataforma. Asegurarse de que la interfaz sea intuitiva y fácil de navegar.

  • Pruebas de Rendimiento:

Evaluar el rendimiento de la plataforma al procesar tesis de diferentes tamaños y complejidades. Medir los tiempos de respuesta y la eficiencia del sistema.

  • Pruebas de Seguridad:

Identificar y mitigar posibles vulnerabilidades de seguridad, como la protección de datos de usuario y la prevención de ataques.

  • Pruebas de Integración:

Probar la interoperabilidad de los componentes del sistema y verificar que se comuniquen correctamente.

  • Pruebas de Carga y Estrés:

Evaluar cómo la plataforma maneja cargas de trabajo intensivas y situaciones de estrés. Verificar la capacidad de escalabilidad.

  • Pruebas de Compatibilidad de Navegadores:

Asegurar que la plataforma sea compatible con varios navegadores web, como Chrome, Firefox, Edge, etc.

  • Pruebas de Accesibilidad:

Evaluar la accesibilidad de la plataforma para usuarios con discapacidades visuales o de movilidad.

  • Pruebas de Registro y Autenticación:

Verificar que el proceso de registro y autenticación de usuarios funcione correctamente y de manera segura.

  • Pruebas de Algoritmos de Procesamiento de Lenguaje Natural (NLP):

Evaluar la precisión y eficacia de los algoritmos de análisis de riqueza léxica y conectividad.

  • Pruebas de Retroalimentación y Puntuación:

Verificar que la retroalimentación proporcionada a los usuarios sea clara, relevante y útil para mejorar sus tesis.

  • Pruebas de Almacenamiento de Datos:

Evaluar la capacidad de la plataforma para almacenar y recuperar los resultados de las evaluaciones de los usuarios de manera segura y eficiente.

  • Pruebas de Colaboración:

Evaluar las funciones de colaboración, como compartir resultados y trabajar en proyectos de tesis de manera conjunta.

  • Pruebas de Notificaciones y Recordatorios:

Verificar que las notificaciones y recordatorios se envíen de manera oportuna y sean configurables por los usuarios.

  • Pruebas de Compatibilidad con Dispositivos Móviles:

Asegurarse de que la plataforma sea accesible y utilizable en dispositivos móviles.

  • Pruebas de Cumplimiento de Regulaciones de Datos:

Garantizar que la plataforma cumple con regulaciones de protección de datos, como GDPR u otras leyes de privacidad.

  • Pruebas de Colisión y Errores:

Identificar y corregir posibles errores, fallos y colisiones que puedan surgir durante el uso de la plataforma.

Tailoring
Impact of not havingNot having developer tests can inhibit iterative development, because there is no assurance that modified components are still working correctly when you modify components iteration by iteration.
Reasons for not needingIf the tests can be embedded into the actual production code, you might not need a separate work product. Nonetheless, some level of support for developer testing is always necessary.
Representation Options

The following are recommendation and options for representing this work product.

Recommendation: Automated Code Unit

The most appropriate technique for running these tests is using code that tests the components fully and that you can run regularly as you update the system during development.

When code is the sole form of the tests, you must take care to ensure that the code is self-documenting, including specifications of what conditions you are testing and what setup or clean-up is required for the test to run properly.

Option: Manual Instructions

In some cases, manual instructions can suffice. For example, when testing a user interface, a Developer could walk through a script, explaining the component. In this case, it can still be valuable to create a test harness that goes straight to the user interface. That way, the Developer can follow the script without having to walk through a complicated set of instructions to get to a particular screen or page.

Option: Embedded Code

Certain technologies (such as Java™ 5 Test Annotation) enable you to embed tests in the implementation. In those cases, there will be a logical work product, but it will be assimilated into the code that you are testing. Here, too, take into consideration that you must ensure that the code is self-documenting.
