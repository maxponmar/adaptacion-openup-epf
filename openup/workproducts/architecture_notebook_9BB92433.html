Artifact: Architecture Notebook
documento centralizado que recopila y registra información relevante sobre la arquitectura de software del proyecto. Contiene detalles esenciales sobre las decisiones arquitectónicas, patrones de diseño, componentes clave y consideraciones técnicas.
Domains: Architecture

Purpose

Proporcionar un recurso que documente y explique la arquitectura de software del proyecto de manera coherente. Este artefacto facilita la comprensión, colaboración y mantenimiento de la arquitectura a lo largo del ciclo de vida del proyecto.

Description
Main Description

1. Aplicación Web RPA Hosteada en GitHub

  • Interfaz de Usuario: Es el punto de contacto entre el usuario y la aplicación. Esta interfaz es crucial para proporcionar una experiencia de usuario fluida y eficiente. Al estar alojada en un servidor de GitHub, se beneficia de la robustez y fiabilidad que ofrece esta plataforma, además de facilitar la gestión del código y las actualizaciones.
  • Servidor GitHub con URL Custom: El alojamiento en un servidor de GitHub proporciona una base sólida para la aplicación web. El uso de una URL personalizada mejora la identidad de marca y la facilidad de acceso para los usuarios.

2. APIs (Backend)

  • API de OAuth (Servicio de Terceros): Esta API maneja el proceso de autenticación y autorización, proporcionando un método seguro y eficiente para gestionar el inicio de sesión de los usuarios. Utilizar un servicio de terceros para OAuth ayuda a mantener la seguridad de los datos de los usuarios y reduce la carga de desarrollo y mantenimiento en esta área crítica.
  • API de Procesamiento de Lenguaje Natural y AI (Python): Desarrollada en Python, esta API realiza tareas de procesamiento de lenguaje natural e inteligencia artificial. Utiliza herramientas como Freeling y un modelo BERT para la evaluación de conectividad, lo que permite analizar y entender el contenido de las tesis de manera eficiente y precisa.
  • API de Lógica de Aplicación (C#): Esta API, desarrollada en C#, gestiona toda la lógica de la aplicación. Es responsable de comunicarse con la base de datos PostgreSQL para administrar el historial de evaluaciones de usuario, resultados y demás información relevante. El uso de C# para esta API asegura un rendimiento robusto y la capacidad de manejar complejas operaciones lógicas.

3. Base de Datos PostgreSQL:

  • Datos de Evaluaciones y Resultados: Una base de datos PostgreSQL almacena y gestiona todos los datos relacionados con las evaluaciones de usuarios, resultados y otra información pertinente. PostgreSQL es una elección excelente para este propósito debido a su escalabilidad, robustez y soporte para complejas consultas y operaciones de datos.


Relaciones y Flujos:

  • Flujos entre la Aplicación Web y las APIs: La aplicación web interactúa con la API de OAuth para la autenticación del usuario, con la API de Procesamiento de Lenguaje Natural y AI para la evaluación de contenido de tesis, y con la API de Lógica de Aplicación para procesar y almacenar los resultados de las evaluaciones.
  • Comunicación entre las APIs y la Base de Datos: La API de Lógica de Aplicación interactúa con la base de datos PostgreSQL para almacenar y recuperar datos relacionados con las evaluaciones de los usuarios y sus resultados.

Justificación de la Arquitectura

Esta arquitectura está diseñada para ser robusta, escalable y segura, con cada componente desempeñando un papel específico que contribuye al funcionamiento eficiente del sistema. La separación de preocupaciones, evidente en la utilización de APIs dedicadas para diferentes tareas, asegura que el sistema sea mantenible y extensible, mientras que el uso de tecnologías consolidadas y confiables garantiza la estabilidad y el rendimiento.


Brief OutlineEl Architecture Notebook es un recurso valioso para los equipos de desarrollo de software, ya que garantiza que la arquitectura sea bien comprendida y que las decisiones arquitectónicas sean coherentes y rastreables a lo largo del proyecto. Ayuda a mantener la integridad y la calidad de la arquitectura del software.

Key Considerations
  • El Architecture Notebook debe mantenerse actualizado a medida que evoluciona el proyecto y se toman nuevas decisiones arquitectónicas.
  • Es importante que el documento sea claro y accesible para todos los miembros del equipo, facilitando la comprensión y la colaboración.
  • La documentación de la arquitectura es fundamental para el mantenimiento a largo plazo del software y para facilitar futuras expansiones o modificaciones.
Tailoring
Representation Options

The architecture can be represented in many forms and from many viewpoints, depending on the needs of the project and the preferences of the project team. It need not be a formal document. The essence of the architecture can often be communicated through a series of simple diagrams on a whiteboard; or as a list of decisions. Choose the medium that best meets the needs of the project.

The architecture can be expressed as a simple metaphor or as a comparison to a predefined architectural style or set of styles. It may be a precise set of models or documents that describe the various aspects of the system's key elements. Expressing it as skeletal build is another option - although this build may need to be baselined and preserved to ensure that the essence of the system can be understood as the system grows.

Part of the architecture frequently references architecturally significant portions of the Deseño. It can reference models that describe Architectural Views for communicating the architecture. A view is a representation of a system from the perspective of a related set of concerns. To choose the appropriate set of views, identify the Stakeholders who depend on software architecture documentation and the information that they need.
