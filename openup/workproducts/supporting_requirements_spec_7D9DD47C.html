Artifact: Especificación de Requisitos de Soporte
Describe los requisitos y las necesidades relacionados con el mantenimiento, soporte y operación continua de un sistema de software una vez que ha sido implementado y puesto en producción.

Purpose
Garantizar que el sistema de software pueda operar de manera confiable y eficiente en su entorno de producción. Proporciona una guía detallada para el equipo de soporte, operaciones y mantenimiento, y establece expectativas claras para la gestión del sistema a lo largo de su ciclo de vida.

Description
Main Description

La Especificación de Requisitos de Soporte incluye los siguientes aspectos clave:

  • Requisitos de Mantenimiento: Describe los requisitos específicos relacionados con las actividades de mantenimiento del sistema, como correcciones de errores, actualizaciones de seguridad y mejoras de rendimiento. Esto puede incluir la frecuencia y el procedimiento para aplicar parches y actualizaciones.
  • Requisitos de Soporte Técnico: Especifica los requisitos para proporcionar soporte técnico a los usuarios finales, incluyendo canales de comunicación, tiempos de respuesta y niveles de servicio acordados.
  • Requisitos de Copias de Seguridad y Recuperación: Detalla los procedimientos y políticas relacionadas con la realización de copias de seguridad de datos críticos y la recuperación de sistemas en caso de fallos.
  • Requisitos de Monitoreo y Gestión de Rendimiento: Describe cómo se monitoreará el sistema en producción para detectar problemas de rendimiento y disponibilidad. También puede incluir umbrales de alerta y métricas clave a seguir.
  • Requisitos de Seguridad y Cumplimiento: Especifica los requisitos relacionados con la seguridad del sistema y el cumplimiento de normativas y estándares relevantes. Esto puede incluir políticas de acceso, autenticación, autorización y auditorías.
  • Procedimientos de Respuesta a Incidentes: Define los procedimientos que se deben seguir en caso de incidentes de seguridad, fallos graves o interrupciones del sistema. Esto incluye la asignación de roles y responsabilidades.
  • Actualización y Evolución del Sistema: Detalla los procesos para la gestión de versiones, actualizaciones de software y planificación de la evolución del sistema a lo largo del tiempo.

Key Considerations
  • La Especificación de Requisitos de Soporte es esencial para garantizar la continuidad operativa del sistema y minimizar el tiempo de inactividad no planificado.
  • Debe ser un documento actualizado que se revisa y ajusta periódicamente para reflejar los cambios en las necesidades operativas y de soporte.
Tailoring
Impact of not having

The goal of this work product is to make sure that all types of requirements are covered, which reduces the risk of not considering some important facet of the system. FURPS+ requirements are system-wide, and they influence the Architectural Mechanisms that you will create, thus guiding development of the system's foundation. These requirements are frequently the major cost item, because they determine architectural choices.

Furthermore, if you do not capture system-wide requirements in a central location, but repeat them throughout the Use Cases, the result will be more maintenance and more chance for error.

Representation Options

Recommendation: Use the Supporting Requirements Specification Template

The Supporting Requirements Specification template provides a tool to capture, structure, and organize the supporting requirements.

Even in a small project, a requirements management tool, a database, or a spreadsheet, are recommended for prioritizing and managing requirements. If Stakeholders are comfortable with accessing requirements directly from that tool, or with accessing a report automatically generated from the tool, you do not need a separate document.

Option: Use the Work Items List

Consider capturing Supporting Requirements in the Work Items List, which you can use for prioritizing and managing requirements. If Stakeholders are comfortable with accessing requirements directly from the work items list, or with accessing a report automatically generated from it, then you do not need a separate document.

Option: Include as Part of the Vision Document

Consider including some types of Supporting Requirements in the Visión del Proyecto document. To keep the Vision stable, follow this option for the requirements types that need less refinement, such as Product Qualities, Documentation, or Compliance.
