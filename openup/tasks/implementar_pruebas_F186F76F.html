Task: Implementar Pruebas de Desarrollo
Implementar una o más pruebas que permitan la validación de los componentes de software individuales mediante su ejecución.
Disciplines: Development
Purpose
Preparar la validación de un componente de software (por ejemplo, una operación, una clase, un procedimiento almacenado) mediante pruebas unitarias. El resultado son una o más nuevas pruebas de desarrollo.

        Main Description

        Las pruebas de desarrollo son diferentes de otras formas de pruebas, ya que se basan en el comportamiento esperado de las unidades de código en lugar de basarse directamente en los requisitos del sistema.

        Es mejor realizar estas pruebas a una pequeña escala, mucho más pequeña que el código completo que un desarrollador producirá durante el transcurso de una iteración. Esto se puede hacer para una operación, un campo agregado a una interfaz de usuario, un procedimiento almacenado, etc. A medida que la base de código se construye de manera incremental, se crearán nuevas pruebas y las pruebas existentes podrían ser revisadas para probar comportamientos adicionales.

        Steps
        Planificar Pruebas de Desarrollo
        En esta etapa inicial, se define el plan de pruebas de desarrollo. Esto incluye identificar los objetivos de las pruebas, los recursos necesarios, los plazos y los criterios de aceptación.
        Diseñar Casos de Prueba
        Diseñar casos de prueba detallados que cubran diferentes aspectos del software, incluyendo funcionalidad, rendimiento, seguridad y usabilidad. Cada caso de prueba debe tener entradas específicas, acciones a realizar y resultados esperados.
        Entornos de Prueba
        Preparar los entornos de prueba, que pueden incluir servidores de prueba, bases de datos de prueba y configuraciones específicas. Asegurarse de que los entornos sean lo más parecidos posible al entorno de producción.
        Implementar Pruebas Automatizadas
        Desarrollar y configurar pruebas automatizadas cuando sea posible. Las pruebas automatizadas son útiles para repetir pruebas con regularidad y para tareas repetitivas.
        Ejecutar Casos de Prueba
        Ejecutar los casos de prueba diseñados en el entorno de prueba. Registrar los resultados de cada prueba, incluyendo cualquier error encontrado.
        Evaluar Resultados de Pruebas
        Analizar los resultados de las pruebas y compararlos con los resultados esperados. Identificar y documentar cualquier error o discrepancia encontrada.
        Corregir y Retestear
        Corregir los errores identificados durante las pruebas y volver a ejecutar las pruebas afectadas para asegurarse de que los problemas se han solucionado correctamente.
        Pruebas de Regresión
        Realizar pruebas de regresión para asegurarse de que las modificaciones realizadas para corregir errores no hayan introducido nuevos problemas en el software existente.
        Pruebas de Rendimiento
        Realizar pruebas de rendimiento para evaluar cómo se comporta el software bajo diferentes cargas y condiciones de uso. Identificar y resolver cuellos de botella de rendimiento.
        Pruebas de Seguridad
        Realizar pruebas de seguridad para identificar posibles vulnerabilidades y riesgos de seguridad en el software. Corregir cualquier problema de seguridad encontrado.
        Documentar Resultados
        Documentar todos los resultados de las pruebas, incluyendo los casos de prueba, los errores identificados, las correcciones realizadas y las conclusiones.
        Informe de Pruebas
        Preparar un informe de pruebas que resuma los resultados de las pruebas y proporcione una evaluación de la calidad del software en desarrollo.
        New StepAprobación para Implementación
        Obtener la aprobación para la implementación del software en producción una vez que las pruebas hayan demostrado que cumple con los estándares de calidad establecidos.
        Seguimiento Post-Implementación
        Realizar un seguimiento de la calidad del software en producción después de su implementación para identificar y abordar cualquier problema que pueda surgir.
        Key Considerations
        1. Automatizar las pruebas mediante una herramienta de pruebas de regresión unitaria (por ejemplo, xUnit) para que los desarrolladores puedan ejecutar pruebas cada vez que realicen cambios en el código.
        2. Realizar pruebas en función del riesgo del componente. Por ejemplo, cuanto más crítico sea un componente, más importante será probarlo exhaustivamente.
        3. Trabajar en conjunto con el Tester en todas las etapas de esta tarea para obtener información sobre las consideraciones de prueba y calidad.
        Alternatives

        Depender de las pruebas de aceptación para validar su software probablemente sea un enfoque que consuma tiempo, se realice tarde en el proceso y no sea tan efectivo como las pruebas de desarrollo para identificar errores y encontrar su ubicación en el código.

        Las pruebas de aceptación son importantes para validar que el software cumple con los requisitos y expectativas del usuario final, pero no deben ser la única forma de pruebas. Las pruebas de desarrollo, realizadas a nivel de código por los propios desarrolladores, son esenciales para identificar y corregir problemas de manera temprana y eficaz. Combinar pruebas de desarrollo y pruebas de aceptación en un enfoque integral de aseguramiento de la calidad suele ser la mejor práctica para garantizar un software de alta calidad.
